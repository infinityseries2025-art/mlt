# MLT Esports Platform — план работы и прогресс

Живой документ. Обновляется по мере работы — отмечаю `[x]` то, что сделано,
`[ ]` то, что ещё нет. Не удалять между сессиями — по нему продолжаем работу.

БД: **Firebase Firestore** (+ Firebase Auth для логина).
**Firebase Storage не используется** — на новых проектах он требует платный
план Blaze, поэтому проект сделан на стандартном бесплатном пакете **Spark**:
- лого команды — пользователь выбирает файл, он сжимается на клиенте до
  маленькой картинки и сохраняется как base64 прямо в документе команды
  в Firestore (без отдельного бакета);
- обложка турнира / фото новости — админ вставляет обычную ссылку (URL)
  на картинку, размещённую где угодно (imgur, свой хостинг и т.п.).

## 1. Firestore-схема (утверждено, менять только всем документом сразу)

```
users/{uid}
  email, nickname, role: "user" | "admin",
  teamId: string|null           // ссылка на свою команду (если создана)
  createdAt

teams/{teamId}
  name, logoUrl, ownerUid,          // logoUrl — base64 data URL (без Storage), не HTTP-ссылка
  main: [ {nick, realName?}, ... ]     // 5 основных
  reserve: [ {nick, realName?}, ... ]  // до 2 запасных
  rating, winrate, trophies, form: ["W","L",...] (последние 5)
  mapWinrate: { [mapName]: {wins, losses} }
  createdAt

players/{playerId}                     // профиль игрока = запись состава команды
  nick, teamId, weapon,
  kd, adr, kast, rating              // агрегаты, пересчитываются после матча
  achievements: [string]
  formGraph: [number]                // рейтинг за последние матчи
  matchIds: [matchId]                 // для «матчи игрока»

tournaments/{tournamentId}
  name, period, prizePool, status: "upcoming"|"live"|"done",
  coverUrl,                           // обложка турнира — обычная внешняя ссылка (URL), без Storage
  regOpen: boolean,
  registeredTeamIds: [teamId],
  bracket: {...}                      // структура сетки (раунды/матчи)
  winnerTeamId

matches/{matchId}
  tournamentId, teamA, teamB, format: "BO1"|"BO3"|"BO5",
  startAt (timestamp), status: "upcoming"|"live"|"finished",
  streamUrl: string|null              // ссылка на трансляцию — НОВОЕ
  score: [a,b] | null,
  maps: [ {name, score, winnerTeamId,
           playerStats: { [playerId]: {k,d,a,dmg,adr,kast,rating} } } ]

news/{newsId}
  title, excerpt, body, tag, imageUrl,   // imageUrl — обычная внешняя ссылка (URL), без Storage
  publishedAt

teamRegistrations/{regId}             // заявки команд на турнир
  tournamentId, teamId, contactInfo, status: "pending"|"approved"|"rejected",
  createdAt

messages/{id}                         // как на InfinitySite — обращения формы обратной связи (если нужно)
```

Автоматика (клиент/облачная функция при подтверждении результата матча):
- пересчёт `players.*` (K/D, ADR, KAST, rating) из `maps[].playerStats`
- пересчёт `teams.form`, `teams.winrate`, `teams.mapWinrate`
- топ-5 игроков турнира = агрегация `playerStats` по всем матчам с этим `tournamentId`, сортировка по `rating`, автоматически, без ручного ввода

Таймер → Live: Cloud Functions с расписанием нужны план Blaze (платный).
На плане Spark (бесплатный) делаем проще: при открытии страницы матчей/турнира
клиент сам проверяет `startAt <= now && status === "upcoming"` и обновляет
статус на `"live"` в Firestore (у первого посетителя, кто зайдёт после старта).
Хватает для реального продакшена такого масштаба.

### Известные упрощения текущей версии (см. также низ `firestore.rules`)
- Первый когда-либо зарегистрированный пользователь автоматически становится
  админом (без Cloud Functions, т.к. они требуют платный план Blaze).
  **Зарегистрируйте себя как администратора сразу после первого запуска**,
  до того как ссылку на сайт увидит кто-то ещё.
- Данные в Firestore пока пустые (в проекте `mlt-site` ещё ничего нет).
  В Админ-панели есть кнопка «Импортировать демо-данные» — она доступна
  только админу и только один раз (если коллекция команд ещё пуста),
  наполняет базу тем же демо-контентом, что был захардкожен в шаблоне.
- Форма «ссылка на трансляцию» у матча — единственное место в админке,
  где остался простой `prompt()` (это один текстовый вопрос, отдельная
  модалка ради одного поля не даёт выигрыша); всё остальное — уже
  полноценные модалки.
- Редактор турнирной сетки (bracket) — это ручной ввод раундов/матчей
  админом (текстовые названия команд + счёт), без автоматического
  «продвижения» победителя в следующий раунд и без привязки к реальным
  teamId. Это осознанное упрощение: авто-прогрессия сетки — заметно
  более сложная логика (нужно однозначно решать, как определять
  победителя, как обрабатывать переигровки и т.п.), и добавлять её
  сейчас — это отдельная следующая итерация, если она понадобится.
- Storage не используется, поэтому: (1) лого команды хранится как сжатая
  base64-картинка прямо в документе Firestore — предел ~300 КБ на поле,
  что с большим запасом покрывает клиентское сжатие до ~5-15 КБ;
  (2) обложка турнира и фото новости — это просто внешняя ссылка (URL),
  которую вводит админ, а не загрузка файла. Если понадобится настоящая
  загрузка файлов для этих двух полей — это Storage и план Blaze, либо
  внешний сервис хранения картинок с загрузкой через их API.

## 3. Готово для использования

Файл `mlt-site.html` теперь самодостаточный — подключается напрямую к
Firebase (`mlt-site` проект) через CDN-модули, никакого сборщика не нужно.
Используется только стандартный бесплатный пакет **Spark**: Firestore + Auth.
Firebase Storage не задействован вообще — картинки не нужно ни включать,
ни настраивать в консоли. Отдан только `firestore.rules` — его нужно
вставить в консоль Firebase (Firestore → Rules) и нажать Publish, иначе по
умолчанию Firestore либо всё запрещает, либо (если оставили тестовый режим)
всё разрешает — оба варианта нежелательны. Файл `storage.rules` больше не
нужен, его можно не использовать.

Порядок первого запуска:
1. В консоли Firebase вставить `firestore.rules`, Publish. Раздел Storage
   трогать не нужно — сайт им не пользуется.
2. Открыть сайт, зарегистрироваться первым — этот аккаунт станет админом.
3. В Админ-панели нажать «Импортировать демо-данные» (по желанию) — или
   сразу начать вносить свои команды/игроков/матчи.
4. Дальше — обычная работа: создание команд пользователями, заявки на
   турниры, ввод результатов матчей админом (статистика пересчитывается
   автоматически).
5. Когда готовы опубликовать сайт — папка `deploy/` уже собрана
   (`firebase.json`, `.firebaserc`, `public/index.html`), инструкция в
   `deploy/DEPLOY.md`: `firebase login`, затем `firebase deploy --only hosting`
   из папки `deploy/`. Это единственный шаг, который нужно выполнить у себя
   локально — он требует входа в ваш Google-аккаунт.

## 4. Статус: MVP полностью готов

Все пункты Фазы A и Фазы B закрыты. Админка полностью на нормальных
модалках (не `prompt()`, кроме одного текстового поля «ссылка на
трансляцию»), есть рабочий редактор турнирной сетки, весь бэкенд —
на бесплатном плане Firebase Spark, без Storage. Из осознанных
упрощений остались только те, что описаны в разделе 2 выше
(бутстрап первого админа, ручная сетка без авто-прогрессии) — это
уже не «недоделки», а сознательные архитектурные решения для MVP
такого масштаба; расширять их стоит только если реально понадобится.
